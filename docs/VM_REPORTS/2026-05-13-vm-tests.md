## Issues Found

### Issue 1 — Services Not Pre-Running (cold-start)

monitor.py and http.server were not running at session start. This is not a code bug — it
means the VM rebooted or no autostart mechanism is in place.

**Recommendation:** See recommendation below about systemd user services or tmux autostart.

### Issue 2 — pane-state.json and window-names.json delayed until tmux exists

When tmux has no active sessions, monitor.py writes only 3 of 5 producer files.
`pane-state.json` and `window-names.json` do not appear until after a tmux session is created.
This is expected behavior (no panes = nothing to write) but worth noting for health checks:
a check that requires all 5 files should first verify tmux is running.

### Issue 3 — opencode --help triggers SQLite migration on first run

On a fresh VM or after a bun upgrade, `opencode` blocks for ~10–15s doing DB migration
before doing anything useful. Tests that use `opencode` should add at least a 15s sleep
after first launch.

### Issue 4 — window-names.json stale entry after kill-window

After killing `octest` window, `window-names.json` still contained `"gmux:1": "octest"`.
The stale entry persists until the next aggregate worker cycle (up to 10s). Not a bug,
but consumers of this file should handle stale entries gracefully.

---

## Recommendations for Host-Side AI

1. **Add autostart for monitor.py + http.server**: Consider a tmux session startup script
   or systemd user service so that services survive reboots without manual intervention.
   Suggested: `~/.config/fish/config.fish` hook or a `scripts/vm-autostart.sh` called from
   the deploy script.

2. **Expose a `GET /health/all` endpoint** that reports the status of all 5 producer files
   (existence + age). This would make remote health-checking from the host trivial.

3. **Document the opencode first-run migration delay** in VM_PROTOCOL.md so test scripts
   know to budget 15s on first opencode invocation.

4. **Consider a `/api/state?include_stale=false` option** that filters out windows that
   no longer exist in tmux, to prevent stale window-names entries from confusing dashboards.

5. **UI path documentation**: The http.server serves the project root, so multiple UI paths
   work (`:5550/ui/v3/index.html` and `:5550/app/src/dashboard/index.html` both return 200).
   VM_PROTOCOL.md only documents `/ui/v3/index.html` — consider adding a note about the
   dashboard UI path.
